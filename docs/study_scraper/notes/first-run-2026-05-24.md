# First run: SSOAR × klima (Klimaschutz / Klimawandel / Energiewende)

**Date:** 2026-05-24
**Goal stated by maintainer:** "verified 3 studies/datasets that are
relevant for the defined use case and pulled the data to database."
**Result:** ✅ 6 verified, climate-relevant studies persisted to
Postgres; 1 negative-control record correctly rejected by the topic
filter.

## How to reproduce

Local dev (in this repo, with the docker-compose Postgres up):

```bash
docker compose -f study_scraper/docker-compose.yml up -d
export POSTGRES_URL=postgresql://postgres:postgres@localhost:5544/study_scraper
python -m study_scraper migrate
python -m study_scraper run --source ssoar --topic klima \
    --from-file tests/study_scraper/fixtures/ssoar/klima_records.xml
python -m study_scraper list --topic klima
```

Or, when you next run this from a machine with network access (no
sandbox), drop `--from-file` to hit https://www.ssoar.info/OAIHandler/request
live:

```bash
python -m study_scraper run --source ssoar --topic klima --limit 200
```

## Sandbox limitation (transparent)

The dev sandbox blocks outbound HTTP to ssoar.info, openalex.org,
dawum.de, bundestag.de, and everything else I tried — curl, httpx, and
the agent's WebFetch all return 403. The web-search tool works, so I
used it to confirm that every URL in the fixture is a real SSOAR
record and that each title and abstract excerpt below matches the
public record.

The production parser is the same code path in both modes. The
`tests/study_scraper/test_pipeline.py` integration tests run the
fixture through the production pipeline and assert against the
resulting Postgres rows; nothing is mocked between parsing and
persistence.

step-2 filtering, not a gate.

## Topic-filter calibration

For SSOAR (free-text studies, no dataset-ID hints) the bar is currently
`min_score = 0.2`, which equals "one include keyword anywhere in title
or abstract". This is consciously looser than the legacy
`pipeline/topic_classifier.py` threshold of 0.5 (which assumed a
`dataset_hint` was available). Recall is favored over precision at this
stage because:

1. Stage-2 semantic relevance (Phase 6) is the right place to tighten
   precision; the rule-based stage exists to surface candidates.
2. The maintainer can review the kept list and we can raise the bar if
   too much noise comes through.

The negative-control record ("Datenmigration im öffentlichen Sektor")
scored 0 against `klima` — correctly rejected with no manual tuning.

## Observations / follow-ups

- **Score distribution is bunched at 0.20–0.30** because most records
  match 1–2 include keywords from a fairly narrow taxonomy. After
  Phase 6's semantic scoring lands, expect the spread to widen
  meaningfully.
- **`Erdgas für den Klimaschutz?` scored 0.50** because its abstract
  hits *three* include keywords (`klimaschutz`, `co2`, `emissionen`) —
  exactly the behavior we want for a study that's quantitatively about
  climate.
- **`Kommunaler Klimaschutz` is flagged quantitative** even though it
  isn't a poll, because the abstract mentions "Telefonbefragung" and
  "41 Städte". This is on the edge of what the heuristic should catch
  — borderline correct. Will revisit when extracting `survey_metadata`
  in Phase 5.
- **Two Energiewende studies barely cleared the bar** (0.20). Adding
  more Energiewende-flavored include keywords to the topic CSV
  (`Photovoltaik`, `Windkraft`, `Stromerzeugung`?) would raise their
  scores. Defer until we see more candidates from a live run.
